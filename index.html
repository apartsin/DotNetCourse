About this course

This is the advanced course on designing and building enterprise-grade backend systems with C# and modern .NET. Students move beyond isolated programming exercises into the engineering discipline of domain modelling, use-case design, API boundaries, persistence, validation, security, testing, architecture documentation, and maintainable code. The emphasis is on building systems whose design can be explained, reviewed, tested, extended, and defended.

No prior .NET experience is assumed. Students arrive with programming, basic OOP, database, and web foundations. The course builds the .NET foundations early, then uses one running project to introduce architecture styles, enterprise design patterns, robust API design, security, testing, and AI API integration through explicit application boundaries.

Rationale. Enterprise software is not only about making endpoints respond. It is about aligning business use cases, domain rules, persistence, external integrations, quality attributes, and team-readable code. This course provides a shared engineering base for students who will build backend systems, distributed applications, AI-enabled services, and capstone projects in realistic organizational settings.

Course format

Each week is 4 contact hours: a 2-hour lecture for concepts, architecture, and design reasoning, and a 2-hour practice session at the keyboard. There are no separate weekly labs: each practice session closes with a project-integration brief, the increment every team adds to its semester system that week. The thirteen weeks form six parts: C# and .NET Foundations (weeks 1 to 2), API and Persistence (3 to 4), Architecture and Patterns (5 to 8), Domain Logic and Security (9 to 10), Testing and AI Integration (11 to 12), and Final Architectural Defense (13). The running project is presented to the class three times, in the Student Project Presentations: a proposal (week 5), an interim review (week 8), and a final project defense (week 13).

AI usage

Using an AI assistant is encouraged in this course; it reflects how modern software engineering is increasingly done. Two conditions keep the learning genuine: students keep full ownership of, and responsibility for, everything they submit, and must be able to explain and defend any part of it. Where an assistant was used, it should be disclosed in the project notes.

Every weekly project increment follows a three-part model:

Part A · AI assistant welcomeBuild

Use an assistant to explore framework APIs, draft tests, generate examples, compare designs, or find refactoring options.

Part B · verification requiredTest & inspect

Run the code, read it, check edge cases, remove secrets from prompts, and make sure generated code matches the architecture.

Part C · in plain languageReview & defend

Explain the design and its tradeoffs, where it would fail, and why each pattern or boundary belongs in the system.

Assessment and grading

Grading is project-based, with weight on architecture, reasoning, tests, maintainability, use-case coverage, and the ability to defend design decisions. The running project is the single deliverable, built up in weekly increments and graded at the three Student Project Presentations. There are no written exams and no separate lab submissions.

The running project

100% of the grade · teams of three or four · presented three times

Teams carry a single enterprise application across the semester, integrating most of the covered material into one coherent system: a business domain, use cases, REST API, persistent data model, layered or Clean Architecture structure, enterprise design patterns, security, validation, tests, documentation, and an optional AI-backed use case. Teams choose one of the course domains, order management, course registration, clinic appointments, helpdesk ticketing, inventory management, booking and reservation, portfolio management, or insurance claims, or propose another domain of comparable scope for approval. Whatever the domain, teams must demonstrate architectural maturity, not endpoint count alone. Each presentation is 12 to 15 minutes plus questions, with a short written report and a tagged release of the repository.

P1Student Project Presentation 1 · Proposal

20% of the grade · week 5 · problem, domain, and architecture plan

  • Project domain, motivation, users, and main use cases.
  • Initial domain model, database entities, and API surface.
  • Planned architecture style and expected design patterns.
  • Main technical risks and development plan.

P2Student Project Presentation 2 · Interim

30% of the grade · week 8 · working foundation and architecture review

  • Working API demo with database integration.
  • Current architecture, layer boundaries, and code structure.
  • Implemented design patterns and core business workflows.
  • Open issues, next steps, and architectural tradeoffs.

P3Student Project Presentation 3 · Final

50% of the grade with a short oral defense · week 13 · complete system

  • Complete working system with a final architecture walkthrough.
  • Main design decisions, enterprise patterns, and use-case coverage.
  • Security, validation, tests, quality assurance, and documentation.
  • Optional AI integration design or implementation, including boundaries and evaluation.
  • Short oral defense: any team member answers on any part of the system.

Tools and resources
